# Scraper lessons

Append here when a real failure or user correction surfaces a recurring
pitfall. Static tool behavior and recipes do **not** belong here — those
live in `CLAUDE.md`. A lesson is a bug we've actually hit.

Format:

```
- **<short title>** — <what went wrong>. <what to do next time>.
```

- **Per-row DB commits after paid external calls** — 2026-04-23 a `run-extraction` batch of 42 URLs burned 210 Firecrawl credits when one CHECK-constraint violation (`subcategory='other'` existed in the Pydantic enum but not the DB CHECK) rolled back the single wrapping transaction, stranding every paid scrape as still-`pending`. When a function pays an external provider *then* writes to the DB, never wrap the writes in one transaction — let each row commit independently so a single bad row only fails itself.
- **Python enum and DB CHECK must stay in lockstep** — adding a value to `SUBCATEGORY_TO_CATEGORY` / the `HairProductSubcategory` Literal without the matching migration lets bad rows pass Pydantic and fail at write time. When editing the enum, always add (or update) the corresponding migration in the same change.
- **URL-encoded unicode can fool the Grok classifier** — 2026-04-24 `filter-links` correctly skipped `microfiber-hair-towel` but kept `daily-hydro%E2%84%A2-hair-towel` (same brand, same `-hair-towel` suffix, only difference is the TM symbol encoded into the slug). Always spot-check the `keep` list for accessory/tool keywords (`towel`, `brush`, `comb`, `filter`) before staging — the skip side is not the only place false classifications hide.
- **Shopify `/collections/all` can be JS-rendered; fall back to sitemap** — 2026-04-24 innersensebeauty.com returned only 1 product URL from `list-page-links` on `/collections/all?limit=250&page=N`, and `/products.json` returned a schema dump (typed shape descriptors, not real JSON). Fetching `/sitemap_products_1.xml` via curl and regex-extracting `<loc>…/products/…</loc>` yielded all 113 products. When a Shopify listing returns <5 products and `products.json` looks malformed, try the sitemap before parking the brand.
- **Seeded `website_url` can belong to an unrelated business** — 2026-04-24 jamaican-mango-and-lime had `jmlbeauty.com` recorded, which is actually a nail/lash/waxing salon. `list-site-urls` returned only services pages; `list-page-links` on `/collections/all` returned 0 product links. Real domain is `jamaicanmangolime.com` (one word, no "and"). When discovery returns 0 products and the site's own URLs look off-brand (services, bookings, unrelated categories), do a `scrape-page` on the root to confirm the site matches the brand before parking — the website column may just be wrong.
- **Seeded domain can 301 to a rebranded storefront** — 2026-04-24 `melaninhaircare.com` 301-redirects to `www.melaninhaircareevents.com` ("Melanin Haircare Events"). `list-page-links` on the old `/collections/all` returned `[]` because Firecrawl doesn't follow cross-domain redirects for listing. Direct `curl -I` on the root reveals the redirect target; scrape the new domain's `/collections/all` via raw HTML + regex on `href="/products/..."` when Firecrawl returns empty. Always resolve redirects before concluding a Shopify brand has gone dark.
- **Seeded domain can be wrong TLD (DNS SERVFAIL)** — 2026-04-24 naturalicious had `naturalicious.com` seeded, but that apex SERVFAILs (no A record). Real domain is `naturalicious.net`. When `list-page-links`/`list-site-urls` both return a `DNS resolution failed` error, don't park the brand — do a `WebSearch` for "<brand name> official website" to find the correct TLD, then retry discovery. This is distinct from the JML case (wrong domain resolves to unrelated business) because here DNS itself fails.
- **Mixed-catalog Firecrawl hallucination needs ≥3 probes** — 2026-04-24 ORS (orshaircare.com) had 1 probe pass cleanly (2% absent) but 2 more probes either hallucinated 88-100% or returned `inci_len=0` because only the newer product line exposes INCI as text — older lines are image-only. The runbook's "if any probe passes, continue" rule is unsafe for brands with split rendering: the validator only checks ≥5 tokens, not markdown-absence, so hallucinated rows land in the DB as `success`. When the first probe passes but you suspect mixed rendering (brand has multiple product lines / different packaging eras), run a 3rd-4th probe across lines before continuing — if ≥50% hallucinate, park the brand.
- **Seeded domain can be entirely unrelated name (not just wrong TLD)** — 2026-04-24 taliah-waajid had `taliahwaajid.com` seeded (SERVFAIL, no DNS). Real DTC store is `naturalhair.org` — the brand's eponymous domain doesn't exist; they operate under a generic category domain. This is beyond the naturalicious `.com→.net` TLD case. When `dig` returns SERVFAIL on the seeded apex, `WebSearch "<brand> official website DTC"` — don't assume the fix is a TLD swap; the canonical domain may share no tokens with the brand name.
- **Per-line pruning beats all-or-nothing park** — 2026-04-24 rizos-curls had 13/15 probes pass cleanly, but both products in the "Biotin" sub-line (biotin-shampoo, biotin-conditioner) hallucinated 91-98% because those pages expose a marketing glossary under "Ingredients" ("Biotin: strengthens hair…") instead of an INCI list — actual INCI only appears on product images. Don't park the whole brand when the hallucination is confined to an identifiable subset: probe a representative URL per line, drop the image-only line from the keep list, and proceed with the remainder. Separately, Firecrawl can persistently return `md_len=0` on a specific URL (deep-conditioner, ~666KB page — likely timing out) while still hallucinating plausible INCI; drop any URL that returns empty markdown across ≥2 probes.
- **`/collections/all` can be a curated subset, not the full catalog** — 2026-04-24 arey.com's `/collections/all` returned 25 products, but `/collections/shop-all` returned 33 (including the flagship Wash Shampoo, Smooth Conditioner, and Masque). The Shopify default `/collections/all` isn't guaranteed to include every active product — merchants can hand-curate it or exclude items. When discovery on `/collections/all` returns suspiciously few products relative to `list-site-urls` (which surfaced e.g. `arey-wash-shampoo`, `arey-smooth-conditioner`, `masque-deep-conditioner` that weren't in `/collections/all`), try `/collections/shop-all` and per-category collections (`/collections/shop-shampoo-conditioners`, etc.) before concluding the catalog is small. Union the product URLs across them.
- **Same-template pages can split null vs hallucinated INCI** — 2026-04-24 burtsbees.com baby-bee shampoos all use the identical PDP template (same "See full ingredient list" widget, INCI only in product image JPEGs). Probes 1–2 honestly returned `ingredient_text=null`; probes 3–4 on sibling URLs of the same template returned fully fabricated INCI lists with 95-100% of tokens absent from markdown. The validator's ≥5-token rule would have landed those hallucinations as `success`. Rule: when one probe returns `null` and another on the same template returns a full INCI list, don't assume the second is real — diff the tokens against markdown. If the site hides INCI in images site-wide, Firecrawl's behavior is random per-call (sometimes honest, sometimes made-up); park the whole brand.
- **Seeded domain can be a GoDaddy parked lander** — 2026-04-24 `betternotyounger.com` (no hyphen) served a 115-byte JS redirect to `/lander`, which loaded a GoDaddy parking page (`window._trfd.push({ap:"parking"})`, `img1.wsimg.com/parking-lander/…`, `LANDER_SYSTEM="PW"`). `list-site-urls` returned only `/` and `/lander`; `list-page-links` on `/collections/all` returned `[]` because that path itself is just the JS-redirect stub. Distinct from wrong-TLD / unrelated-business / cross-domain-301 cases — the apex resolves and serves 200s, it's just been repossessed and parked. Real brand lives on the hyphenated variant `better-notyounger.com`. When `list-site-urls` surfaces only 1–2 URLs and one is `/lander`, `curl` the root and grep for `parking` / `wsimg.com/parking-lander` before parking the brand in the DB — then `WebSearch` for the correct domain.
- **Grok classifier false negatives on generic-noun bundle slugs** — 2026-04-24 Better Not Younger had `/products/volume-fullness` and `/products/volume-hydration` pass `filter-links` as products, but the actual `<title>` reveals both are bundles ("…Shampoo, Conditioner & Hair Thickener Bundle"). Distinct from the TM-unicode lesson: these slugs contain zero bundle/kit/duo tokens — the classifier has no lexical signal. Spot-check the keep list for any slug that's two abstract generic nouns joined (`volume-X`, `hydration-Y`, `strength-Z`) on brands that visibly push bundles; `curl` the page and grep `<title>` to confirm single-product before staging.
- **Fabricated Latin binomials are a hallucination tell on botanical-heavy brands** — 2026-04-24 Kama Ayurveda (Ayurvedic botanical brand, INCI only in product images) had 2/4 probes return plausible-looking INCI lists that included invented Latin names like `TUBER ARUM (ARROWROOT)` (real INCI for arrowroot is `MARANTA ARUNDINACEA`) and `PHALAKKAHARA (MARSHMALLOW)` (not a genus — marshmallow root is `ALTHAEA OFFICINALIS`). The other 2 probes honestly returned `null`. When auditing probe extractions on Ayurvedic / botanical / herbal brands, don't just diff tokens against markdown — also sanity-check binomials: if a Latin name looks non-standard (genus not Title-Case single word, family name in parentheses, or common-name-first order) treat it as hallucination even if the markdown absence check is inconclusive. The 5-token validator cannot catch this and will land the fabricated row as `success`.
- **B2B salon brands publish zero public INCI by design** — 2026-04-24 Shiseido Professional's real domain (`shiseido-professional.com`, with hyphen — the unhyphenated `shiseidoprofessional.com` is an unrelated marketing-agency squatter) serves product-detail pages that honestly return `ingredient_text=null`, and `curl | grep -iE "(ingredient|inci|aqua|water|sodium)"` on the raw HTML returns zero matches. Distinct from image-only INCI (Burt's Bees, Rizos Biotin) — the ingredient data is not on the public site in any form (image or text) because the brand is salon-distribution only and treats formulations as stylist-facing info. No product pages carry price or currency either. When the site has no DTC e-commerce affordances (no cart, no price, "Find a salon" CTA instead of "Add to bag"), confirm with one raw-HTML grep for INCI markers before spending more preflight credits; if zero hits site-wide, park immediately.
- **Featured-ingredient footnotes defeat token-absence diffs** — 2026-04-24 Ichikami (Kracie corporate catalog, `kracie.co.jp/eng/products/ichikami/`) publishes per-product marketing footnotes that list curated "hero" ingredients (e.g. "Pure Japanese Botanicals Premium Extract: Perilla, Rice Bran, Cherry Blossom, Calendula, Ashitaba, Blackberry Lily, Soapberry, Camellia" + a separate "cleansing ingredients" footnote: "Lauroylsarcosine TEA, TEA-Cocoyl Glutamate"). Firecrawl stitched these footnotes together with a leading "Water," into a plausible 12-token "INCI" that passed the 5-token validator AND returned only 16.7% markdown-absence — every token genuinely appears in the page, just not as a real INCI list. A sibling probe on the same template (Premium Mask) hallucinated 66.7% (CETEARYL ALCOHOL, BEHENTRIMONIUM METHOSULFATE etc. invented outright); a third probe returned `null`; a fourth returned `md_len=0`. The real INCI is image-only on packaging. New defense beyond markdown-diff: when a brand's markdown has bulleted ingredient-callout footnotes (`*N: <comma list>` or "Formulated with X, Y, Z"), treat extracted INCI as suspect even at low absence — diff against at least 2 sibling probes and require one of them to independently produce the same INCI. If one sibling returns null or <50% overlap, the low-absence "success" was just a footnote-stitch, not a real formulation.
- **imweb.me sites: `/all/?idx=N` is the product URL, not a list page** — 2026-04-24 Rated Green (ratedgreenglobal.com) runs on imweb.me (Korean site builder). Product URLs look like `/all/?idx=100`, `/haircare/?idx=92` — a category path + `?idx=<N>` query. `list-page-links` returns only nav links (page is JS-rendered); raw `curl | grep -oE '/[a-z]+/\\?idx=[0-9]+'` extracts the real product URLs, and `/all?page=1..N` paginates. `filter-links` (Grok) classifies every one of these as `"reason": "list page"` because `?idx=` looks like pagination syntax — you get 0 kept / N skipped even though all are valid PDPs. When discovery yields `/all/?idx=N`-style URLs on an imweb site, skip `filter-links` or manually override its output. Separately, Rated Green itself parks for the usual reason (image-only INCI: all 4 probes returned null `ingredient_text` with no aqua/water/sodium tokens in markdown) — the lesson here is the discovery + filter pattern, not the park decision.
- **"Key Ingredients" marketing callout + hallucination co-occur in K-beauty catalogs** — 2026-04-24 Moremo (moremo.com, Korean salon brand) every PDP has a `## Key Ingredients` section listing 5–12 common-name items ("Brazil Nut, Olive, Argan, Almond, Camellia, Macadamia, Avocado, Jojoba" — common botanical names, not INCI). Two of four probes (Hair Essence, Caffeine Biome Shampoo) passed the 5-token validator AT 0% markdown-absence because the callouts genuinely appear in the page, yet they are NOT an INCI list. The other two probes (Repair Shampoo, Miracle 10) produced fully fabricated INCI at 85% / 99% absence. Raw-HTML grep for `aqua|water,|sodium laureth|cocamidopropyl|inci` returned zero matches site-wide — the site simply doesn't publish INCI. New defense: if probe-1 "INCI" is <15 tokens AND tokens are all common names (no `-eth`, no `-trimonium`, no `dimethicone`, no parenthetical Latin binomials), treat it as a marketing callout regardless of absence %, and probe deeper. Then raw-HTML grep the domain for INCI markers; zero hits → park. Applies to Shopify K-beauty brands where the product page only exposes curated hero ingredients.
- **JS-stub HTML + identical-template marketing callouts yield per-probe INCI divergence** — 2026-04-24 Oshima Tsubaki (`ost.oshimatsubaki.co.jp/en/`, J-beauty corporate catalog, 8 SKUs) serves ~1.9KB of raw HTML per PDP (pure JS stub — `curl | grep -iE 'aqua|water,|sodium|ingredient'` returns zero across every URL). Every product uses an identical template with exactly 4 Camellia hero-ingredient image callouts ("Camellia oil (moisturizing agent) blended", etc.). Probe 1 (Premium Shampoo) returned a 5-token INCI ending in `...` — already a truncation tell — with `CAMELLIA OLEIFERA SEED OIL`, `CAMELLIA OLEIFERA SEED CERAMIDE`, `CAMELLIA JAPONICA FLOWER EXTRACT` (plausible binomials absent from markdown). Probe 2 (Premium Conditioner, same template) returned a completely disjoint 11-token list — `Cetearyl Alcohol, Behentrimonium Methosulfate, Crambe Abyssinica Seed Oil, Acid Blue 9, Benzyl Alcohol` — zero overlap with probe 1. Distinct from ichikami (which stitches real footnote tokens) and moremo (common names only): here Firecrawl invents different plausible INCI per call on pages that share bytes-identical marketing copy. New tell: when ≥2 probes on pages with the same 4-callout template produce INCI lists with <20% token overlap, Firecrawl is generating per-request, not extracting — park immediately regardless of validator pass, without spending more preflight credits.
- **Raw-curl B2B check false-negates on SPA storefronts; Firecrawl render decides** — 2026-04-24 Milbon USA (`www.milbon-usa.com`, iBeauthentic CMS) raw-curl returned 2.7KB stubs with zero hits for `aqua|water|sodium|cocamidopropyl|dimethicone|ingredient` site-wide — identical signal to the shiseido-professional "park immediately" rule. But Firecrawl's JS-rendered `scrape-page` on a shampoo PDP revealed full public INCI paragraphs (`WATER, SODIUM LAURETH SULFATE, CYCLOPENTASILOXANE, DIMETHICONE, POLYQUATERNIUM-52, AMODIMETHICONE`, Latin binomials like `SAXIFRAGA SARMENTOSA EXTRACT`, `POLIANTHES TUBEROSA POLYSACCHARIDE`), 4 preflight probes all at 0% markdown-absence with 19–39 real INCI tokens. The site is B2B (login-walled pricing, "Log in to view pricing!" on every PDP, no cart, no consumer price) — salon-distribution ≠ hidden INCI. Final job: 63 success / 15 missing / 0 failed of 78 staged; missing rows were in-salon "professional treatment" and color-system pages that genuinely expose only hero ingredients. Refinement to the shiseido-professional rule: "zero raw-HTML INCI hits → park" is only valid when the PDP is ≥20KB server-rendered and still lacks INCI markers. When the PDP body is ≤5KB and looks SPA-shaped (empty main, JS bundles only), raw-curl's zero-hit signal is meaningless — spend one `scrape-page` (5 credits) on a likely-INCI product before parking.
- **Enumerate catalog via category + brand-lineup union when no sitemap exists** — 2026-04-24 Milbon USA has no `sitemap.xml` (404), its `/site_map` page renders only category-level links (no product URLs), and `list-site-urls` surfaced just 22 product URLs across the domain. `list-page-links` on each `/categories/<X>` index returned 62 unique products; unioning with `/brands/milbon/<line>` pages (moisture, smooth, repair, anti-frizz, volume, scalp, liscio, enhancing_vivacity, illuminating_glow, etc.) added 21 more → 83 total. Both axes were individually incomplete because the merchant curates which products appear in each index. Extends the arey.com "union per-collection pages" lesson: when a site has no `/collections/all`-equivalent AND no XML sitemap AND `list-site-urls` returns <30 products, enumerate both the merchandising-category axis and the brand-sub-line axis, then union.
